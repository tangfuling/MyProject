## 4. Chrome 插件技术方案

### 4.1 文件结构

```
extension/
├── manifest.json          # Manifest V3 配置
├── background.js          # Service Worker，管理登录态
├── content-script.js      # 注入微信后台页面，数据采集
├── popup.html             # 弹窗界面
├── popup.js               # 弹窗逻辑
└── icons/                 # 插件图标
```

### 4.2 manifest.json 关键配置

```json
{
  "manifest_version": 3,
  "name": "公众号数据运营助手",
  "permissions": ["activeTab", "storage"],
  "host_permissions": [
    "https://mp.weixin.qq.com/*",
    "https://api-gzh.niuma.com/*"
  ],
  "content_scripts": [{
    "matches": ["https://mp.weixin.qq.com/*"],
    "js": ["content-script.js"]
  }],
  "background": {
    "service_worker": "background.js"
  },
  "action": {
    "default_popup": "popup.html"
  }
}
```

### 4.3 数据采集流程

```
content-script.js 注入 mp.weixin.qq.com
        │
        ▼
 渲染悬浮按钮（页面右下角）
        │
        ▼ 用户点击
 提取当前页面 token（URL 参数）
        │
        ▼
 Step 1: 获取已发布文章列表
   GET /cgi-bin/appmsgpublish?sub=list&begin=0&count=10&token={token}
   → 分页遍历，提取所有文章 ID、标题、发布时间
        │
        ▼
 Step 2: 获取每篇文章阅读数据
   GET /misc/appmsganalysis?action=detailpage&msgid={mid}_1&publish_date={date}&token={token}
   → 提取：阅读、分享、点赞、在看、评论、收藏、完读率、来源分布
        │
        ▼
 Step 3: 获取文章全文（新文章）
   GET /s/{article_id}（带 cookies）
   → 提取正文纯文本，去 HTML 标签
        │
        ▼
 Step 4: 发送到后端
   POST https://api-gzh.niuma.com/sync/articles
   Body: { articles: [...], snapshots: [...] }
```

### 4.4 关键实现细节

- **Cookie 自动携带**：Content Script 在微信域下运行，fetch 请求自动携带 cookie
- **Token 提取**：从当前页面 URL 的 `token` 参数中获取
- **增量同步逻辑**：
  - 通过 `chrome.storage.local` 记录上次同步的文章 ID 列表
  - 新文章：全量抓取（列表 + 数据 + 全文）
  - 已有文章：只更新数据指标（追加 snapshot）
- **进度通知**：通过 `chrome.runtime.sendMessage` 向 popup 发送进度更新

### 5.2 后端架构约束

| 约束项 | 说明 |
|--------|------|
| **Controller 禁止写业务逻辑** | Controller 仅做参数接收、调用 Service、返回结果 |
| **Service 是唯一事务边界** | `@Transactional` 只能放在 Service 层 |
| **对象分层强制隔离** | `DTO` 仅入参，`Entity` 仅持久化，`VO` 仅出参，禁止混用 |
| **参数校验强制开启** | 所有入参必须使用 `@Valid` + `jakarta.validation` 注解 |
| **统一鉴权上下文** | 用户身份统一从 `AuthContext` 获取，禁止在业务代码重复解析 token |
| **分页查询必须显式分页** | 列表接口必须有分页参数，禁止全量查询后内存分页 |
| **写操作必须幂等** | 创建/支付等接口必须具备业务幂等键 |
| **TraceId 全链路透传** | 每个请求必须具备 traceId，日志按 traceId 可串联 |
| **错误码统一治理** | 错误码由 `common` 统一管理，禁止模块私定义 |

### 6.1 SSE 流式输出封装

前端通过 `EventSource` / `fetch + ReadableStream` 实现流式文本接收，实时渲染 AI 输出。

```ts
// common/network/SseClient.ts
type SseCallbacks = {
  onChunk: (content: string) => void;
  onDone: (data: Record<string, unknown>) => void;
  onError: (error: Error) => void;
};

export function createSseStream(
  url: string,
  body: Record<string, unknown>,
  token: string,
  callbacks: SseCallbacks
): AbortController {
  const controller = new AbortController();

  fetch(url, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${token}`,
    },
    body: JSON.stringify(body),
    signal: controller.signal,
  })
    .then(async (response) => {
      const reader = response.body!.getReader();
      const decoder = new TextDecoder();
      let buffer = '';

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        buffer += decoder.decode(value, { stream: true });
        const lines = buffer.split('\n');
        buffer = lines.pop() || '';

        for (const line of lines) {
          if (!line.startsWith('data: ')) continue;
          const json = JSON.parse(line.slice(6));
          if (json.type === 'chunk') callbacks.onChunk(json.content);
          if (json.type === 'done') callbacks.onDone(json);
        }
      }
    })
    .catch((err) => {
      if (err.name !== 'AbortError') callbacks.onError(err);
    });

  return controller;
}
```

### 7.2 核心接口详情

#### POST /auth/send-code

```json

{ "phone": "13800001234" }

// Response
```

- 验证码存 Redis，TTL 5 分钟
- 频率限制：同一手机号 60 秒内不可重发

```json
      "balance": 100,
```

## 9. AI 引擎设计

### 9.1 多模型适配架构

系统支持多个 AI 模型提供商，通过统一接口 + 工厂模式实现模型热切换。

```
┌────────────────────────────────────────────┐
│              AiClient（统一接口）            │
│  + streamChat(messages, tools) → SseEmitter │
│  + getModelName() → String                  │
│  + calcCost(tokensIn, tokensOut) → int      │
└───────────┬────────────────────────────────┘
            │
  ┌─────────┼─────────┬──────────┐
  ▼         ▼         ▼          ▼
千问      豆包      Claude     Codex
(Qwen)   (Doubao)  (Anthropic) (OpenAI)
```

```java
// common/ai/AiClient.java
public interface AiClient {
    /** 流式对话，返回 SseEmitter */
    SseEmitter streamChat(List<ChatMessage> messages, List<ToolDef> tools);
    /** 模型标识 */
    String getModelName();
    /** 计算费用（分） */
    int calcCost(int tokensIn, int tokensOut);
}

// common/ai/AiClientFactory.java
@Component
public class AiClientFactory {
    @Resource private QwenAiClient qwenClient;
    @Resource private DoubaoAiClient doubaoClient;
    @Resource private ClaudeAiClient claudeClient;
    @Resource private CodexAiClient codexClient;

    public AiClient getClient(String model) {
        return switch (model) {
            case "qwen" -> qwenClient;
            case "doubao" -> doubaoClient;
            case "claude" -> claudeClient;
            case "codex" -> codexClient;
            default -> qwenClient;
        };
    }
}
```

### 9.3 分层喂数据策略

```
第一层：全量概览（每次必传）
  所有文章：标题 + 发布时间 + 字数 + 阅读/分享/收藏/完读率/来源占比
  → 100 篇 ≈ 3,000-5,000 tokens
  → 模型看到全局趋势、对比排序

第二层：选定范围全文（按用户选择的时间范围）
  如近 30 天的 8 篇文章全文
  → 8 篇 × 1,500 字 ≈ 8,000-12,000 tokens
  → 模型分析内容质量、风格、钩子、结构

第三层：对话时按需加载（function calling）
  用户提到具体文章时，模型请求加载那篇全文
  → 单篇 ≈ 1,500-2,000 tokens
```

### 9.4 对话 Context 构成

```
每轮对话传入：
├── System Prompt（角色设定 + 分析框架）      ≈ 500 tokens
├── 全量概览（始终带着）                      ≈ 4,000 tokens
├── 当前分析报告（已生成的）                   ≈ 1,500 tokens
├── 按需加载的文章全文                         ≈ 0-2,000 tokens
└── 对话历史                                  ≈ 变量
```

### 9.5 分析报告 Prompt 框架

```
你是一个公众号数据运营分析师。以下是用户公众号的全部数据。

## 全量概览
{全量概览 JSON}

## 选定范围文章全文
{文章全文列表}

请输出以下结构的分析报告：

1. **你现在在什么阶段**：用大白话说清楚，禁止使用"冷启动期""内容验证期"等术语
2. **核心发现**：3-5 条，分正向（✓）和负向（✗），必须引用具体数据
3. **可执行建议**：3 条，每条必须是本周能做的具体动作
4. **节奏感**：基于数据事实给出理性鼓励，不鸡汤

最后，输出 5 个推荐问题，放在 recommended_questions 字段。
要求：基于这个用户的数据生成，口语化，≤15 字，让用户想点。

输出格式：Markdown 正文 + 末尾 JSON 块包含 recommended_questions。
```

## 11. 计费引擎

### 11.1 计费流程

```
每次 AI 调用：
1. 预检查：余额 + 免费额度 > 0，否则拒绝
2. 调用完成后，从 AI 模型响应获取实际 token 用量
3. 通过 AiClient.calcCost() 计算费用（不同模型定价不同）
4. 扣费顺序：优先扣 free_quota，不足部分扣 balance
5. 写入 token_logs 记录明细（含 ai_model 字段）
6. 更新 users 表余额
7. 以上步骤在数据库事务中完成（@Transactional + 乐观锁）
```

### 11.2 费用计算

```
cost = aiClient.calcCost(tokens_in, tokens_out)

各模型按各自 API 定价换算为人民币（分），结果四舍五入。
不同模型单价差异较大，用户切换模型时可在设置页查看各模型定价说明。
```

## 12. 安全设计

| 风险 | 措施 |
|---|---|
| JWT 泄露 | Token 有效期 30 天，支持服务端吊销 |
| 短信轰炸 | 同一手机号 60 秒限频，同一 IP 每小时 10 次 |
| 数据越权 | 所有 API 通过 JWT 中的 user_id 限定数据范围（AuthContext 统一获取） |
| 支付安全 | 支付宝异步回调使用签名验证，金额与订单匹配 |
| 余额扣费 | 数据库事务 + 乐观锁，防止并发超扣 |
| AI Key 泄露 | API Key 通过环境变量注入，禁止硬编码在代码或配置文件中 |
| 微信接口变更 | 预留 CSV 手动上传降级方案（V2） |
| 接口安全 | 参数校验强制开启（@Valid），统一异常处理，禁止返回堆栈信息 |
