# 插件部署文档

## 1. 文档目标
本文档用于说明 `gzh-extension` 的本地测试部署与发布部署方式，明确浏览器兼容性，以及 `ZIP` 与 `CRX` 两种分发方式的区别和建议。

## 2. 插件目录
- 插件根目录：`/Users/ke/Documents/Mine/Works/MyProject/gzh-extension`
- 说明：当前插件为原生 Chrome Extension（Manifest V3），无前端构建步骤，不需要 `npm run build`。

## 3. 浏览器兼容性

### 3.1 可直接兼容（Chromium 内核）
- Google Chrome
- Microsoft Edge
- Brave
- Opera
- Vivaldi
- 其他支持 Manifest V3 + 开发者模式加载扩展的 Chromium 浏览器

### 3.2 需适配后兼容
- Firefox：需做 MV3 细节适配（API、权限、行为差异）
- Safari：需通过 Apple 工具链转换与签名，不能直接使用当前目录

## 4. 本地测试部署（推荐）

### 4.1 推荐方式
本地开发与测试，**建议直接“加载已解压扩展程序”**，无需打包。

### 4.2 操作步骤
1. 打开浏览器扩展页：`chrome://extensions`（Edge 为 `edge://extensions`）。
2. 开启“开发者模式”。
3. 点击“加载已解压的扩展程序”。
4. 选择目录：`/Users/ke/Documents/Mine/Works/MyProject/gzh-extension`。
5. 安装后可在扩展列表看到“公众号数据运营助手”。

### 4.3 联调使用步骤
1. 启动后端服务（确保 `/sync/articles` 可访问）。
2. 启动 Web 前端并登录（例如 `http://localhost:5173`）。
3. 在插件弹窗中填写：
   - API Base：本地联调通常为 `http://127.0.0.1:8081`
   - Auth Token：Web 登录后的 JWT（`localStorage` 中 `gzh_token`）
4. 打开并登录 `https://mp.weixin.qq.com`。
5. 点击插件弹窗“开始同步”或页面右下角悬浮按钮开始同步。

### 4.4 开发迭代
- 修改插件代码后，在扩展页点击“重新加载”即可生效。

## 5. 发布部署

## 5.1 版本管理要求
发布前必须更新 `manifest.json` 的 `version` 字段（例如 `1.0.0` -> `1.0.1`）。

## 5.2 ZIP 打包（源码分发）
用于交付源文件包、存档、或提交某些平台审核。

```bash
cd /Users/ke/Documents/Mine/Works/MyProject/gzh-extension
zip -r ../gzh-extension-v1.0.0.zip . -x "*.DS_Store"
```

说明：
- `ZIP` 通常不能被浏览器“直接安装运行”为已安装扩展。
- 接收方一般仍需先解压，再“加载已解压扩展程序”。

## 5.3 CRX 打包（安装包分发）
适用于内部离线分发与安装包形态交付。

### 5.3.1 通过 Chrome 打包
1. 打开 `chrome://extensions`
2. 开启“开发者模式”
3. 点击“打包扩展程序”
4. 扩展程序根目录选择：`/Users/ke/Documents/Mine/Works/MyProject/gzh-extension`
5. 私钥文件：
   - 首次打包可留空（系统会自动生成 `.pem`）
   - 后续版本必须复用同一个 `.pem`

### 5.3.2 输出文件
- `gzh-extension.crx`
- `gzh-extension.pem`（私钥）

### 5.3.3 私钥管理要求
- `.pem` 是发布资产，必须妥善保存。
- 若私钥丢失，后续包会变更扩展 ID，导致无法平滑升级。

## 6. ZIP 与 CRX 的区别

| 维度 | ZIP | CRX |
|---|---|---|
| 形态 | 源码压缩包 | 浏览器扩展安装包 |
| 是否可直接安装 | 通常不行（需解压后加载） | 可作为扩展包分发（受浏览器策略限制） |
| 适用场景 | 代码交付、归档、审查 | 内部分发、安装包交付 |
| 更新一致性 | 无内建机制 | 依赖固定私钥可保持同一扩展 ID |
| 推荐程度 | 本地开发不推荐依赖 ZIP | 发布交付推荐 CRX |

## 7. 建议方案（结论）
- 本地测试：**直接加载已解压目录（最推荐）**。
- 对外发布/交付：**打包 CRX（并保管 PEM）**。
- 代码归档/走审：补充提供 ZIP。

## 8. 常见问题

### 8.1 为什么本地不建议先打包再测？
直接加载目录可以“改完即 reload”，调试效率最高。

### 8.2 CRX 能否长期更新？
可以，但必须一直使用同一个 `.pem`。

### 8.3 为什么在某些浏览器安装 CRX 受限？
部分浏览器对非商店来源 CRX 有策略限制，可改用“企业策略分发”或“加载已解压目录”。
